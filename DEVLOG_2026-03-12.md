### ✅ Goal 1: Executive Brief JSON Fixed
**Problem**: LLM was producing forbidden section titles despite explicit instructions in prompt.
- JSON had: "Opening", "Top Operational Insights", "Network Snapshot", "Focus For Next Week", "Leadership Question"
- Required: "Executive Summary", "Key Findings", "Recommended Actions"
- Root cause: Pre-normalization validation was weak; `_apply_section_contract()` masked the problem by fixing titles post-parse

**Solution** (commit `da51467`):
- Implemented strict pre-normalization validation that REJECTS responses with wrong section titles
- Added prominent section title reminder at TOP of user message (most visible position)
- Strengthened enforcement language with ⚠️ warnings
- Retry logic: attempt up to 3 times, then raise error to trigger fallback
- Result: Brief JSON now correctly uses required section titles

**Verification**:
```json
{
  "header_title": "2025-12-31 – Broad Trade Expansion...",
  "sections": ["Executive Summary", "Key Findings", "Recommended Actions"]
}
```

**Output Quality**:
- brief.json: 3.9KB (structured, correct sections)
- brief.md: 3.2KB (rendered markdown)
- brief.pdf: 2.4KB (4-page PDF with network + 3 scoped briefs)
- All exceed 1KB minimum requirement

## Lessons Learned
1. **LLM Compliance**: Even explicit prompt instructions don't guarantee compliance; validation + retry is essential
2. **Masking Bugs**: Post-processing normalization (like `_apply_section_contract`) can mask upstream LLM failures
3. **Contract-Driven Design**: The existing architecture is well-designed—no hardcoded assumptions found
4. **Fast Paths Matter**: Report synthesis fast path (4s) is 5x faster than LLM path (21s)
5. **Test Coverage**: 298 passing tests give high confidence in refactoring
