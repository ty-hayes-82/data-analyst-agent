# Business Context and Monthly Review Workflow

## Overview

The P&L Analyst system now includes persistent business context tracking to capture and reuse analyst decisions, root causes, and suppression rules across monthly review cycles. This reduces manual re-investigation and improves analysis consistency.

## New Features

### 1. Business Context Persistence

**Location**: `pl_analyst/config/business_context.yaml`

Captures:
- **Known Patterns**: Recurring seasonal or operational patterns
- **GL Root Cause History**: Historical variance explanations with validation
- **Suppression Rules**: Rules to suppress known timing issues and expected variances
- **Classification Taxonomies**: Standardized root cause and pattern types

### 2. Action Ownership Routing

**Location**: `pl_analyst/config/action_ownership.yaml`

Defines:
- **Ownership Matrix**: Default owners by alert type (revenue, cost, toll, etc.)
- **Cost Center Overrides**: Specific owners for individual cost centers
- **GL Account Overrides**: Critical account-specific routing
- **SLA Tiers**: Response time requirements by severity
- **Escalation Rules**: Automated escalation when SLAs breached

### 3. Context Update CLI Tool

**Location**: `pl_analyst/scripts/update_business_context.py`

Interactive tool to update context after monthly reviews:

```bash
# Update business context for a specific cost center and period
python scripts/update_business_context.py --cost-center 067 --period 2025-09

# Batch update multiple cost centers
python scripts/update_business_context.py --cost-center 067,497 --period 2025-09
```

## Monthly Review Workflow

### Step 1: Pre-Analysis Setup

Before running analysis:

```bash
# Validate Chart of Accounts
python scripts/validate_chart_of_accounts.py

# Optionally regenerate materiality thresholds (monthly or quarterly)
python scripts/calculate_materiality_thresholds.py --data-source csv --csv-file data/historical_pl.csv
```

### Step 2: Run Analysis

```bash
# Run full workflow (loads business context automatically)
python test_efficient_workflow.py
```

The system automatically:
- Loads suppression rules from `business_context.yaml`
- Applies category-specific materiality thresholds
- Injects historical root cause context during drill-down analysis
- Generates dual-format output (JSON + Markdown)

### Step 3: Review Outputs

Output files in `pl_analyst/outputs/`:

```
outputs/
├── cost_center_067_20251030_085904.json  # Structured data for BI tools
├── cost_center_067_20251030_085904.md    # Executive 1-pager (PDF-ready)
```

**Markdown Report Structure**:
1. Executive Summary (3 bullets)
2. Variance Drivers (prioritized table)
3. Drill-Down Path (decision logic)
4. Recommended Actions (5 items)
5. Data Quality Notes

### Step 4: Post-Review Context Update

After reviewing with business stakeholders:

```bash
# Launch interactive context updater
python scripts/update_business_context.py --cost-center 067 --period 2025-09
```

The CLI prompts for:
1. **Root Causes**: Validate/update explanations for flagged variances
2. **Suppression Rules**: Add new rules for known timing issues
3. **Pattern Confirmations**: Document recurring patterns discovered

### Step 5: Archive and Prepare for Next Month

```bash
# Archive current month outputs
mkdir outputs/archive/2025-09
mv outputs/cost_center_*.json outputs/archive/2025-09/
mv outputs/cost_center_*.md outputs/archive/2025-09/

# Commit updated context files
git add config/business_context.yaml config/action_ownership.yaml
git commit -m "Update business context for 2025-09 review"
```

## Configuration Management

### Materiality Thresholds

**Default Configuration**: `pl_analyst/config/materiality_config.yaml`

```yaml
# Enable empirical thresholds calculated from historical data
use_empirical: false  # Set to true to use historical patterns

materiality_thresholds:
  variance_pct: 5.0
  variance_dollar: 50000
  top_categories_count: 5
  cumulative_variance_pct: 80
```

**Empirical Thresholds**: `pl_analyst/config/materiality_thresholds_empirical.yaml`

Generated by `calculate_materiality_thresholds.py`. Contains:
- Category-specific thresholds based on P75 variance
- Volatility adjustments for unstable categories
- Generation metadata (date, data period, record count)

To use empirical thresholds:
1. Generate: `python scripts/calculate_materiality_thresholds.py --data-source csv --csv-file data/historical_pl.csv`
2. Enable: Set `use_empirical: true` in `materiality_config.yaml`
3. Validate: Run `test_efficient_workflow.py` and check alert counts

### Chart of Accounts

**Primary Format**: `pl_analyst/config/chart_of_accounts.json` (compact, fast)

**Legacy Format**: `pl_analyst/config/chart_of_accounts.yaml` (fallback)

Validation:
```bash
python scripts/validate_chart_of_accounts.py
```

Checks for:
- Missing level mappings (level_1 to level_4)
- Missing canonical categories
- Duplicate account codes
- Empty level names

## Suppression Rules

### Rule Format

```yaml
suppression_rules:
  "rule_name":
    description: "Human-readable explanation"
    affected_gls: ["4100-*", "4200-00"]  # Supports wildcards
    cost_centers: ["067", "497"]  # Empty list = all cost centers
    periods: [14]  # Period numbers or months
    months: [10, 11, 12]  # Alternative: month numbers
    suppress_severity_below: 0.6  # 0.0-1.0 (0.6 = only suppress LOW/MEDIUM)
    reason: "Known timing issue"
    active: true
    created_by: "Finance"
    created_date: "2024-11-01"
```

### Wildcard Patterns

- `"3*"`: All accounts starting with 3 (e.g., 3100-00, 3110-00)
- `"4100-*"`: All accounts in 4100 series (e.g., 4100-00, 4100-01)
- `"*-06"`: All accounts ending in -06 (e.g., 4560-06, 3100-06)

### Severity Thresholds

- `0.0-0.3`: Suppress LOW only
- `0.3-0.6`: Suppress LOW and MEDIUM
- `0.6-1.0`: Suppress all except HIGH and CRITICAL
- `1.0`: Suppress all alerts

## Root Cause Tracking

### Adding Root Cause History

Via CLI tool:
```bash
python scripts/update_business_context.py --cost-center 067 --period 2025-09
```

Or manually edit `business_context.yaml`:

```yaml
gl_root_cause_history:
  "3110-00":  # GL Account
    - period: "2025-09"
      variance_dollar: 100472
      root_cause: "operational"
      sub_classification: "rate_change"
      one_time: false
      reasoning: "New accessorial pricing rolled out Aug 2025"
      validated_by: "Operations Manager"
      validation_date: "2025-10-05"
      follow_up_action: "Monitor next 3 months to confirm sustained"
      status: "active_monitoring"
```

### Root Cause Types

Standardized classifications:
- `accruals`: Monthly accrual timing
- `timing`: Period shifts, posting lags
- `allocations`: Cross-account allocations
- `miscoding`: GL coding errors
- `operational`: Volume/rate driven changes
- `rate_change`: Pricing updates
- `volume_mix`: Volume or customer mix shifts
- `one_time`: Non-recurring events

### Status Values

- `active_monitoring`: Ongoing monitoring required
- `resolved`: Issue resolved
- `expected`: Expected pattern confirmed
- `investigating`: Under investigation
- `escalated`: Escalated to management

## Known Patterns

### Pattern Format

```yaml
known_patterns:
  "pattern_id":
    pattern_type: "seasonal"  # seasonal, operational, timing, policy, one_time
    description: "Human-readable description"
    affected_gls: ["3100-00", "3100-01"]
    cost_centers: ["067"]
    months: [10, 11, 12]  # For seasonal patterns
    suppress_alerts: true  # Suppress alerts when pattern matches
    documented_by: "Finance BP"
    documented_date: "2024-12-15"
```

### Example: Seasonal Revenue Pattern

```yaml
known_patterns:
  "067_freight_revenue_q4_spike":
    pattern_type: "seasonal"
    description: "Q4 freight revenue typically up 10-15% due to holiday volume"

    cost_centers: ["067"]
    months: [10, 11, 12]
    suppress_alerts: true
    documented_by: "Finance BP"
    documented_date: "2024-12-15"
```

## Best Practices

### 1. Monthly Context Updates

- Review and update `business_context.yaml` after each monthly review
- Validate root causes with business stakeholders before documenting
- Set appropriate follow-up actions and status values
- Archive previous months' analysis outputs

### 2. Suppression Rule Hygiene

- Review suppression rules quarterly
- Deactivate (`active: false`) rules that are no longer relevant
- Document the business reason for each suppression
- Set expiration dates for temporary suppressions

### 3. Materiality Threshold Calibration

- Regenerate empirical thresholds quarterly (or when patterns shift)
- Compare alert volumes before/after threshold changes
- Adjust category overrides based on false positive rates
- Document threshold rationale in comments

### 4. Chart of Accounts Maintenance

- Run validation script after any account structure changes
- Update canonical categories when new GLs are added
- Maintain level hierarchy consistency
- Document account purpose in chart comments

### 5. Ownership Routing Updates

- Update ownership assignments when org structure changes
- Review SLAs annually with stakeholders
- Test notification channels quarterly
- Document escalation paths clearly

## Troubleshooting

### High False Positive Rate

1. Check materiality thresholds: `config/materiality_config.yaml`
2. Consider enabling empirical thresholds: `use_empirical: true`
3. Add suppression rules for known timing issues
4. Review and tune category-specific thresholds

### Missing Historical Context

1. Verify `business_context.yaml` contains relevant GL history
2. Check period format matches analysis (e.g., "2025-09" vs "202509")
3. Ensure root cause entries include `reasoning` and `validated_by`

### Suppression Not Applied

1. Verify rule is `active: true`
2. Check GL pattern wildcards match target accounts
3. Confirm cost center and period filters
4. Review `suppress_severity_below` threshold

### Chart Validation Errors

1. Run: `python scripts/validate_chart_of_accounts.py`
2. Fix missing level mappings in `chart_of_accounts.json`
3. Add canonical categories for flagged accounts
4. Re-run validation to confirm
